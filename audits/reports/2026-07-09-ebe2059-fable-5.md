## Executive summary

This is a follow-up audit, run immediately after the remediation commit `ebe2059` that responded
to the first audit (`2026-07-09-4f915d4-sonnet-5.md`). The remediation work is substantially real
and correct: this audit independently re-verified, in source, that **15 of the 24 prior findings
are fixed** — including the single Critical (close-to/account-drain fields are now extracted and
rendered as red error warnings on all three signing surfaces) and four of the five Highs
(wallet-at-rest encryption is now PBKDF2-SHA256 (600,000 iterations) + AES-256-GCM with transparent
legacy upgrade; the private-key console dump is gone along with every `console.log` in `src/`;
addresses are tap-to-expand in signing views; WalletConnect peer links are sanitized and carry
`rel="noopener noreferrer"`). One prior High (node genesis trust) is Mitigated by a new
suggested-params cross-check. However, reviewing the new code surfaced **one new High finding**:
`changePassword` leaves the *old* password in session state, so the very next auto-save silently
re-encrypts the wallet under the old password, reverting the change without any error — a
lockout/false-security risk for exactly the users who change passwords after a suspected
compromise. Three new Low findings (legacy obfuscation-grade wrapping of the in-memory session
password, the PBKDF2 derived-key cache surviving logout, and known CVEs in transitive dependencies
— most of them reachable only through the still-present legacy WalletConnect v1 client) and nine
carried-forward Medium/Low findings round out the picture. Recommended next actions: fix
`changePassword` (small, contained change), then retire the WalletConnect v1 dependency chain,
which now accounts for most of the `pnpm audit` noise.

**Findings this audit:** New: 1 High, 3 Low. Carried forward open: 2 Medium-leaning (AW-2026-010,
-011), plus 7 further Medium/Low. Fixed since last audit: 15 (verified). Mitigated: 1.

## Scope & methodology

Performed per `audits/ai-audit-instructions.md` (version `ebe2059`), sections 0–5. Emphasis of
this pass, given the code delta since the prior audit was almost entirely the remediation commit:

## Findings

### AW-2026-025 — `changePassword` silently reverts: old password remains in session state and re-encrypts the wallet on the next auto-save
**Severity:** High · **Status:** New

**Affected files:**
- `src/store/wallet.ts:1366-1407` (`changePassword`), `:1408-1453` (`saveWallet`), `:679-698`
  (`setIsOpen` mutation), `:726-729` (`lastActiveAccount` action → `saveWallet`)
- `src/pages/Settings.vue:468-478` (`changePasswordClick`)

**Description:** `changePassword` verifies the old password by dispatching
`openWallet({ name, pass: passw1 })` (`wallet.ts:1388`), which commits `setIsOpen` and stores the
**old** password (`passw1`), wrapped, into `state.pass`. It then re-encrypts the wallet blob under
the **new** password (`passw2`) and writes it to IndexedDB (`wallet.ts:1404-1405`) — but never
updates `state.pass` to the new password. `saveWallet` derives its encryption password from
`state.pass` (`wallet.ts:1409-1412`) and is dispatched by many routine actions (switching the
active account, `lastPayTo`, adding assets, WalletConnect storage writes, …). The first such
action after a password change re-encrypts the entire wallet with the *old* password and
overwrites the record — the password change is silently undone. Compounding this,
`Settings.vue:470-477` does not `await` the dispatched action and shows the "password updated"
success toast unconditionally, even when `changePassword` returned early on a mismatch or wrong
old password.

**Attacker precondition / trigger:** No attacker needed — normal use triggers it. The user changes
their password and then does anything at all in the same session.

**Impact:** Two distinct harms. (1) **False security:** a user who changes their password because
the old one leaked (shoulder-surfed, reused, phished) continues to have their wallet-at-rest
protected by the compromised password, while the UI told them it was changed. (2) **Lockout /
perceived corruption:** at next unlock the new password fails; a user who deliberately forgot the
old password (believing it replaced) loses access to all key material in that wallet unless they
recall it. Rated High: it undermines the at-rest protection the wallet just shipped (AW-2026-002's
fix) under a completely ordinary user action.

**Recommendation:** At the end of `changePassword`, commit `setIsOpen({ name, pass: passw2 })`
(or a dedicated mutation updating `state.pass`) before returning, and add a regression test-shaped
verification (change password → dispatch any saving action → reopen with the new password). In
`Settings.vue`, `await` the dispatch and only toast success when it returns `true`; on success
consider forcing a re-login so the session provably runs on the new password.

### AW-2026-026 — In-memory session password wrapped with legacy CryptoJS scheme keyed from `localStorage` (obfuscation, not protection)
**Severity:** Low · **Status:** New

**Affected files:** `src/store/wallet.ts:679-698` (`setIsOpen`), `:1409-1412` (`saveWallet`),
`:1630-1647` (`encrypt`/`decrypt` actions), `localStorage` keys `rs1`/`rs2`.

**Description:** While the wallet blob's at-rest encryption was upgraded to PBKDF2 + AES-GCM, the
session mechanism still holds the user's wallet password in Vuex state encrypted with
`CryptoJS.AES.encrypt(pass, rs1)` — the same legacy passphrase overload (single-round-MD5
EVP_BytesToKey) retired elsewhere — where `rs1` is a random string persisted *in `localStorage` of
the same origin*. Any code able to read Vuex state can also read `localStorage`, so this wrapping
adds no security boundary against the only realistic attacker (in-origin script / XSS / malicious
extension); it only prevents the password appearing in cleartext in casual state dumps. `rs2` is
generated and persisted but never read anywhere (`grep` confirms).

**Impact:** No standalone exploit path; this is a hygiene/defense-in-depth observation and an
honest-labeling issue (the code reads as if the password is protected in memory). Interacts with
AW-2026-012 (no auto-lock): the recoverable password sits in memory for the tab's lifetime.

**Recommendation:** Either accept and document it as obfuscation, or hold a session key in
non-persistent memory only (e.g. keep the derived non-extractable `CryptoKey` and drop the
password string entirely after unlock — the PBKDF2 cache in `walletCrypto.ts` already enables
this). Remove the dead `rs2`.

### AW-2026-027 — PBKDF2 derived-key cache is never cleared on logout
**Severity:** Low · **Status:** New

**Affected files:** `src/scripts/encoding/walletCrypto.ts:28-32, 91` (`derivedKeyCache`),
`src/store/wallet.ts:763-766` (`logout` action).

**Description:** Derived AES keys are cached per (password, salt, iterations) for the lifetime of
the tab to keep repeated `saveWallet` calls responsive — a reasonable design, and the keys are
non-extractable `CryptoKey` objects, which is good. However, `wallet/logout` clears passwords and
accounts from Vuex state but never clears this module-level cache, so after logout an in-origin
attacker (XSS, malicious extension) who can also read the IndexedDB blob can still decrypt it via
`subtle.decrypt` with the cached key — without knowing the password — until the tab is closed.

**Recommendation:** Export a `clearDerivedKeys()` from `walletCrypto.ts` and call it from the
`logout` action (and on `destroyWallet`).

### AW-2026-028 — Known CVEs in transitive dependencies; most reachable only via the legacy WalletConnect v1 chain and one entirely unused package
**Severity:** Low · **Status:** New

**Affected files:** `package.json`, `pnpm-lock.yaml`. `pnpm audit --prod` at this commit reports
7 advisories: 3 high, 2 moderate, 2 low.

**Description / breakdown:**
- `ws` 7.x — two high-severity DoS advisories, reachable only via
  `@walletconnect/client` (v1, legacy) → `@walletconnect/socket-transport`. Directly raises the
  cost of keeping AW-2026-024 open.
- `bn.js` infinite loop (moderate) — same WC v1 chain (4 paths).
- `path-to-regexp` ReDoS (high) — via the `page` package, which is **not used at all**: the only
  reference in `src/` is a commented-out import in `src/main.ts:5`. Dead weight with a
  high-severity advisory attached; removing the dependency removes the finding.
- `@stablelib/ed25519` signature malleability (moderate) — via the WalletConnect v2 relay-auth
  chain; used for relay authentication, not for Algorand transaction signing (which is algosdk/
  libsodium), so no direct funds impact identified.
- `elliptic` (low) — via the `crypto-browserify` Node-polyfill shim, not used for wallet key
  operations.
- `esbuild` dev-server file read (low) — devDependency chain, no production exposure.

**Impact:** No identified direct path to key material or signing integrity; DoS-grade at worst in
production code paths. The concentration of advisories in the WC v1 chain is the actionable signal.

**Recommendation:** Remove `page` from `package.json` immediately (unused). Fold the rest into the
AW-2026-024 decision: removing `@walletconnect/client` eliminates the `ws` and `bn.js` advisories
outright.

## Resolved since last audit

All fixed in commit `ebe2059` ("feat: implement wallet encryption and decryption using AES-GCM and
PBKDF2", 2026-07-09). Each was independently re-verified in source during this audit:

- **AW-2026-001 (Critical) — Fixed.** `closeRemainderTo`/`assetCloseTo` now extracted and rendered
  as red `Message severity="error"` warnings with explanatory copy in `Sign.vue` (single:
  `:177-190`, multisig: `:330-341`; extraction `:856-876`), `SignAll.vue` (column `:477-483`,
  expanded row `:530-541`, extraction `:137-143`), and `ConnectRequestsTable.vue` (column
  `:164-170`, expanded row `:217-226`); `store/wc.ts` surfaces `closeTo` on transaction summaries
  (`:31, :291-309`). Localized (10 new keys in all 10 locale files).
- **AW-2026-002 (High) — Fixed.** New `src/scripts/encoding/walletCrypto.ts`: PBKDF2-SHA256,
  600,000 iterations via Web Crypto, AES-256-GCM, 16-byte random salt, 12-byte random IV per
  encryption, format `AWEv1.<iter>.<salt>.<iv>.<ct>`. All four write paths (`createWallet`,
  `saveWallet`, `changePassword`, `encrypt` action) use it; `openWallet` transparently re-encrypts
  legacy CryptoJS blobs on successful open (`wallet.ts:1471-1477`). Design review of the new code
  found the scheme itself sound (authenticated encryption; salt reuse across saves with fresh IVs
  is acceptable; iteration count meets current OWASP guidance) — but see new findings AW-2026-025
  and AW-2026-027 for defects in the surrounding session handling.
- **AW-2026-003 (High) — Fixed.** `grep -rn "console.log" src/` returns 0 matches; the
  `privateAccounts` dump is gone. Remaining `console.error` calls log addresses/errors only.
- **AW-2026-004 (High) — Fixed.** `AlgorandAddress.vue` now toggles the full untruncated address on
  click/Enter (keyboard-accessible, works on touch) plus a native `title` attribute.
- **AW-2026-005 (High) — Mitigated.** `algod.ts:80-104` `assertParamsMatchNetwork` cross-checks the
  node's `genesisID` against the selected network and its `genesisHash` against the known CAIP-10
  prefix, called from all three transaction-building paths (`:172, :261, :271`). Balance/asset data
  from the node is still trusted, and the CAIP-10 reference list is itself unpinned (AW-2026-011) —
  hence Mitigated, not Closed.
- **AW-2026-006 (High) / AW-2026-013 (Medium) — Fixed.** All three WC peer links in `Connect.vue`
  (`:93-99, :176-183, :333-340`) route `href` through `normalizeUrl()` and carry
  `rel="noopener noreferrer"`.
- **AW-2026-007 / AW-2026-014 — Fixed.** `SignAll.vue:467-473, 518-525` renders `txn.rekeyTo`
  (encoded) with the red error treatment; `ConnectRequestsTable.vue:155` gives the WC batch table
  the same emphasis.
- **AW-2026-008 (Medium) — Fixed.** `Connect.vue:416-420, 506-509` reads
  `verifyContext.verified.{validation,isScam}` and renders a valid/invalid/unknown/scam trust badge.
- **AW-2026-009 (Medium) — Fixed.** `Connect.vue:460`: `const allAccounts = ref(false)`.
- **AW-2026-018 (Low) — Fixed.** `ConnectRequestsTable.vue:374-390, 770+` shows a

- **AW-2026-019 (Low) — Fixed.** `getSK` (`wallet.ts:733-761`) is now iterative with a visited-set
  cycle guard; on cycle it logs an error and returns `undefined` instead of recursing.
