## Executive summary

This is the first formal audit of AWallet. AWallet is a client-side, non-custodial Algorand
wallet with no backend; its central risk is whether a user's keys or transaction intent can be
stolen or misrepresented. The review found the wallet's cryptographic *signing* correctness to be
solid ‚Äî every account type (plain, ARC-76, HD, multisig, Ledger, WalletConnect) signs the correct
byte representation, and multisig threshold enforcement is correctly left to consensus rather than
client-side trust. However, the audit identified one **Critical** gap and five **High** severity
issues that materially threaten user funds under realistic conditions with no unusual user action
required: the wallet never displays or warns about `closeRemainderTo`/`assetCloseTo` fields in any
signing surface (a transaction can look like a small payment while silently draining 100% of an
account's balance or ASA holding to an attacker), wallet-at-rest encryption uses a weak,
single-round-MD5 key derivation instead of a slow KDF, full private key material is logged to the
browser console on every account switch, addresses are truncated to 4+4 characters everywhere with
the full value reachable only via a hover tooltip (unusable on mobile), self-originated payments
trust the configured node's genesis data with no cross-check, and a WalletConnect-v1 peer link
renders an unsanitized `href` with no `rel="noopener"`. Twelve Medium and several Low/Informational
findings round out the report, covering session-scope over-granting, missing DApp domain
verification, unpinned public-node-list bootstrap, absent auto-lock, and minor hygiene gaps. None
of the Critical/High findings require a backend fix ‚Äî all are addressable within this repository.
Immediate remediation is recommended for the close-remainder-to display gap and the KDF weakness
before the next release, as both are directly fund-loss-relevant and require no unusual attacker
capability.

**Findings by severity:** Critical: 1 ¬∑ High: 5 ¬∑ Medium: 9 ¬∑ Low: 8 ¬∑ Informational: 8 (33 total,
including positive/no-finding confirmations recorded for auditability).

## Scope & methodology

Performed per `audits/ai-audit-instructions.md` (instruction file `99dfdc1`), sections 0‚Äì5, in
full. Static review of key material handling, signing logic, WalletConnect/DApp surface, chain
data trust, client-side storage, dependency supply chain, build/deployment integrity, web
application hygiene, and UI/UX trust cues in signing flows. Findings were traced across four
parallel focused reviews and then independently re-verified by re-reading the cited source lines

per methodology ¬ß2 item 4 rather than skipped silently. No code was modified during this audit.

## Findings

### AW-2026-001 ‚Äî No warning or display anywhere for `closeRemainderTo` / `assetCloseTo` (account-draining fields)
**Severity:** Critical ¬∑ **Status:** New

**Affected files:**
- `src/store/wc.ts` (`DecodedTransactionSummary` interface and `session_request` handler ‚Äî extracts `rekeyTo` but never `payment.closeRemainderTo` / `assetTransfer.closeRemainderTo`)
- `src/components/ConnectRequestsTable.vue` (summary row and full expansion panel ‚Äî every other transaction field is shown, this one never is)
- `src/pages/Sign.vue`, `src/pages/SignAll.vue` (same omission in the manual/self-initiated and batch review UIs)
- Confirmed repo-wide: `grep -rn "closeRemainderTo|assetCloseTo|CloseTo" src/` returns **zero matches** in the entire application.

**Description:** Algorand payment transactions carry an optional `closeRemainderTo` field (and
asset-transfer transactions an `assetCloseTo` field) that sends the account's *entire remaining
balance* (or entire ASA holding) to a specified address and closes it out, in addition to whatever
explicit `amount` is set. None of the wallet's three transaction-review surfaces ‚Äî the manual
single-transaction confirmation (`Sign.vue`), the batch/scheduled-payment review (`SignAll.vue`),
or the WalletConnect DApp-request review (`ConnectRequestsTable.vue`) ‚Äî read or render this field,
even in their most detailed, fully-expanded views. `signBySk`/other signer paths in
`src/store/signer.ts` sign the transaction bytes exactly as received/constructed, so whatever value
this field holds is signed and broadcast without ever being surfaced to the user.

**Attacker precondition:** A malicious or compromised WalletConnect-connected DApp (session
approval requires only scanning/pasting a pairing URI and clicking Connect ‚Äî no unusual action)
sends a sign request for a transaction dressed up as an innocuous small payment (e.g. "send 0.001
ALGO to X") with `closeRemainderTo` set to an attacker address. Alternatively, a compromised
scheduled-payment configuration or crafted deep link could place the same field in a
self-initiated batch reviewed via `SignAll.vue`.

**Impact:** Complete drain of the account's remaining ALGO balance, or complete loss of a specific
ASA holding, with the wallet's own "review before you sign" UI showing every other field (note,
group, app args, fee, even the rekey warning where present) except the one that matters most here.
This is a direct, realistic path to fund loss via the wallet's normal, expected WalletConnect
signing flow ‚Äî no unusual user action beyond clicking "Sign," which is squarely Critical per this
audit's severity scale.

**Recommendation:** Extract `payment.closeRemainderTo` and `assetTransfer.closeRemainderTo` (algosdk
transaction fields, confirmed present in `algosdk@3.6.0`'s type definitions) in `wc.ts`'s
transaction-summary construction and in `Sign.vue`/`SignAll.vue`'s transaction decoding, and render
a prominent red warning (matching or exceeding the existing `rekeyTo` treatment in `Sign.vue:167-176`)
in both the collapsed and expanded views of all three review surfaces whenever either field is
non-empty. Given the severity, consider gating signing of any transaction with a populated close-to
field behind an additional, harder-to-miss confirmation step.

### AW-2026-002 ‚Äî Weak key derivation for wallet-at-rest encryption (single-round MD5, not a slow KDF)
**Severity:** High ¬∑ **Status:** New

**Affected files:** `src/store/wallet.ts` ‚Äî `createWallet` (~1500-1517), `saveWallet` (1399-1444,
verified), `openWallet` (1445-1470, verified), `checkPassword` (1471-1482), `changePassword`
(1386-1398).

**Description:** Every place the full wallet blob (containing raw `sk` and `hdMnemonic` values for
every account) is encrypted or decrypted calls `CryptoJS.AES.encrypt(data, pass)` /
`CryptoJS.AES.decrypt(encryptedData, pass)` with `pass` as a plain JavaScript string, e.g.
`wallet.ts:1438`: `const dataencoded = CryptoJS.AES.encrypt(data, pass);` and `wallet.ts:1452`:
`const decryptedData = CryptoJS.AES.decrypt(encryptedData, pass);`. When CryptoJS's `AES.encrypt`
receives a raw string rather than a `CipherParams`/`WordArray` key, it falls back to OpenSSL's
legacy `EVP_BytesToKey` scheme: a **single round of MD5** over `password + 8-byte salt` derives the
AES key and IV. This is not a memory-hard or iteration-configurable KDF, and MD5 is fast on
commodity/GPU hardware, making offline brute-force of a weak-to-medium user password far cheaper
than it should be for material this sensitive.

Notably, the same file implements a strong KDF elsewhere (`addEmailPasswordAccount`,
`wallet.ts:921-943`: PBKDF2-SHA256, 999,999 iterations) ‚Äî but only to derive a single ARC-76
account's key from email+password at creation time; once persisted, that account's `sk` is
protected at rest by the weak wallet-level scheme above like every other account.

**Attacker precondition:** Possession of the encrypted wallet blob ‚Äî via the built-in
`backupWallet` export feature, a stolen/cloned device's IndexedDB, or a compromised backup file.

**Impact:** Offline password cracking is dramatically cheaper than intended; a password a user
believes is adequately protecting their entire wallet (all accounts, all mnemonics) may fall to a
motivated attacker with GPU resources far faster than the same password would resist under a proper
KDF.

**Recommendation:** Derive the AES key via PBKDF2 (reusing the already-implemented high-iteration
pattern from ARC-76), scrypt, or Argon2 before passing a derived key/IV pair into
`CryptoJS.AES.encrypt`/`decrypt`, instead of relying on the passphrase-overload's default KDF.

### AW-2026-003 ‚Äî Full private key material (raw `sk`, `hdMnemonic`) logged to the browser console on every account switch
**Severity:** High ¬∑ **Status:** New

**Affected file:** `src/store/wallet.ts:311` (verified):
```js
lastActiveAccount(state, addr: string) {
  console.log("lastActiveAccount set to", addr, state.privateAccounts);
  ...
```

**Description:** This mutation commits on every active-account change, including at normal
wallet-open/session-restore time, and unconditionally logs the entire `privateAccounts` array ‚Äî
raw `sk` byte arrays and `hdMnemonic` strings for every account in the currently open wallet ‚Äî to
the DevTools console. The app does patch `console.log`/`debug`/`warn`/`info` to no-ops in
`config.ts`'s `setConfig` when a fetched `/config.json`'s `debug` flag is falsy, but this
suppression depends on a successful async network fetch completing before any account-switch
mutation fires; Vue 3 does not block child-component mounting on an async `created()` hook, so a
persisted-session restore can plausibly commit this mutation before the console is muted, and the
suppression never activates at all if the `/config.json` fetch fails or is blocked.

**Attacker precondition:** Any channel that can read the page's DevTools console output ‚Äî a
malicious or compromised browser extension with console access, a physically present attacker with
an open DevTools session, or any future console-capturing telemetry/session-replay tooling.

**Impact:** Full exposure of every private key and mnemonic in the open wallet via a channel
entirely independent of, and not gated by, the password-protected Export flow.

**Recommendation:** Remove `state.privateAccounts` from this log line entirely (log only `addr`);
do not rely on a network-fetched flag to suppress logging of secret-bearing data ‚Äî the log
statement should not exist in a form capable of emitting secrets, regardless of debug-flag state.

### AW-2026-004 ‚Äî Addresses truncated to first-4/last-4 characters in every signing confirmation view; full address reachable only via hover (unusable on mobile)
**Severity:** High ¬∑ **Status:** New

**Affected file:** `src/components/AlgorandAddress.vue:14-18, 31` (verified) ‚Äî the single shared
component used to render addresses across `Sign.vue`, `SignAll.vue`,
`ConnectRequestsTable.vue`, and `Connect.vue`:
```js
const shortAddress = computed(() => {
  const addr = props.address ?? "";
  if (addr.length <= 10) return addr;
  return `${addr.substring(0, 4)}..${addr.substring(addr.length - 4)}`;
});
```
```html
<span v-tooltip="props.address ?? ''" class="algorand-address">
  <span class="algorand-address-short">{{ shortAddress }}</span>
```

**Description:** Every recipient, rekey-to, and multisig address shown in every review-before-sign
surface is rendered as e.g. `7ZUE..VDJY` ‚Äî only 4 leading and 4 trailing base32 characters. Algorand
addresses are checksum-suffixed (the last 4 base32 characters are a checksum of the preceding
bytes), so an attacker able to generate a vanity address matching an arbitrary prefix and the
correct checksum-derived suffix (a known, feasible attack ‚Äî vanity-address/address-poisoning
phishing is an established pattern in Algorand and other blockchain ecosystems) produces a string
indistinguishable from the genuine address at this truncation level. The only way to see the full
address is a `v-tooltip` hover, which is easy to skip under signing-flow time pressure and does not
function at all on touch/mobile devices ‚Äî leaving mobile users, a first-class supported client for
this SPA, with no in-flow way to verify a recipient or rekey-to address before signing beyond
copying it out to compare elsewhere.

**Attacker precondition:** Attacker gets a vanity/lookalike address into `payTo`/`rekeyTo` via a
compromised DApp, a poisoned address-book/clipboard entry, or a crafted `algorand://` QR code (which
writes directly into `payTo` per `Sign.vue`'s QR decode path).

**Concrete impact:** Fund theft or account takeover via rekey that a diligent user reviewing the
confirmation screen still cannot reliably catch, because the confirmation UI itself withholds the
information needed to catch it ‚Äî defeating the purpose of the review screen for exactly the two
fields where it matters most.

**Recommendation:** For final signing-confirmation screens specifically, show the full untruncated
address by default (not gated behind hover), or truncate far less aggressively with a visible
tap/click-to-reveal affordance rather than relying on `:hover`. List/table views elsewhere (account
list, transaction history) can reasonably keep the current truncation.

### AW-2026-005 ‚Äî Self-originated payments trust the configured node's genesis/suggested-params data with no cross-check
**Severity:** High ¬∑ **Status:** New

**Affected files:** `src/store/algod.ts` (`getTransactionParams`/`preparePayment`, consumed by
`src/pages/Pay.vue`) ‚Äî confirmed via `grep` that `genesisID`/`genesisHash` are referenced nowhere in
`algod.ts`. By contrast, `src/pages/Sign.vue:1756-1757` (verified) *does* perform this check for
externally-supplied raw signed transactions: `if (txn.value?.genesisID && txn.value.genesisID !==
envName.value) { fatal.value = ... }`.

**Description:** The wallet's own "Pay" flow builds transactions directly from
`algodClient.getTransactionParams().do()` ‚Äî whatever `genesisID`/`genesisHash`/fee/validity window
the currently configured node returns is trusted and signed verbatim, with no comparison against
the genesis data associated with the network the user believes they're on
(`rootState.config.env`/`publicData.getGenesisList`). The existing cross-check pattern in
`Sign.vue` for the WalletConnect/deep-link path is not applied to this self-initiated path at all.

**Attacker precondition:** The wallet is pointed at a malicious or compromised algod node ‚Äî either a
user-entered custom endpoint (see AW-2026-010) or a compromised entry in the auto-selected public
provider list (see AW-2026-011).

**Impact:** A malicious node can return genesis data for a different real network while the UI still
shows the user's intended network name, and can serve fabricated balance/asset data. The wallet
signs a transaction that is fully valid on the real network the malicious genesis actually
corresponds to; the attacker can capture and broadcast it there ‚Äî a cross-network replay/drain that
the wallet's own existing guard (already implemented for the WalletConnect path) does not cover
here.

**Recommendation:** Apply the same genesisID/genesisHash cross-check already implemented in
`Sign.vue` inside `algod.ts`'s payment/transaction-building path, before signing.

### AW-2026-006 ‚Äî Unsanitized `href` (including `javascript:` scheme) and missing `rel="noopener"` on untrusted WalletConnect v1 peer link
**Severity:** High ¬∑ **Status:** New

**Affected file:** `src/pages/Connect.vue:93-100` (verified):
```html
<a target="_blank" class="m-1" :href="slotProps.data.peer.url" :title="slotProps.data.peer.description">
  {{ slotProps.data.peer.name }}
</a>
```

**Description:** `slotProps.data.peer.url`/`.description`/`.name` are metadata self-reported by
whatever DApp/peer initiated a legacy WalletConnect v1 session ‚Äî fully attacker-controlled, with no
validation observed anywhere in the WC1 handling code. Unlike the WalletConnect v2 equivalents in
the same file (`Connect.vue:179, 304`), which route through a `normalizeUrl()` helper that coerces
any non-`http`/`//`-prefixed string to `https://...` (neutralizing a `javascript:` scheme), this WC1
link uses the raw attacker-supplied string directly as `href`, with no `rel="noopener noreferrer"`
either.

**Attacker precondition:** Victim scans/pastes a WalletConnect v1 pairing URI from a malicious or
compromised peer, then clicks the peer name link in the wallet's own Sessions table.

**Impact:** A `javascript:`-scheme `peer.url` can execute script in the wallet's own origin when
clicked in several browsers, independent of `target="_blank"`; given the wallet holds decrypted key
material in memory while unlocked, in-origin script execution is high-impact. Independent of that
edge case, the missing `rel="noopener noreferrer"` on a `target="_blank"` link to a fully
attacker-controlled URL is a standard reverse-tabnabbing vector (the opened page can redirect the
original wallet tab to a phishing clone while the user isn't looking).

**Recommendation:** Route this `href` through the same `normalizeUrl()`/validation already used for
the WalletConnect v2 case, and add `rel="noopener noreferrer"`.

### AW-2026-007 ‚Äî `SignAll.vue` rekey-to column bound to a nonexistent field; always renders blank regardless of actual transaction content
**Severity:** Medium ¬∑ **Status:** New

**Affected file:** `src/pages/SignAll.vue:24-33` (`TransactionTableEntry` interface ‚Äî no `rekeyTo`
property), `:62-70` (row construction ‚Äî never populates one), `:455-459` (column definition:
`<Column field="rekeyTo" ... />` with no `#body` template, so PrimeVue reads
`slotProps.data.rekeyTo`, which is always `undefined`).

**Description:** The batch-sign review table declares a rekey-to column, but the underlying row
objects never carry that field ‚Äî confirmed via a full-file grep for `rekeyTo` matching only the one
dead column definition. This column is permanently blank whether or not the underlying transaction
actually carries a `rekeyTo`. The sibling component `ConnectRequestsTable.vue` (used for
WalletConnect batch review) correctly computes and renders `encodeAddress(slotProps.data.txn.rekeyTo)`
‚Äî so a working pattern exists in the codebase, it just wasn't applied here.

**Attacker precondition:** Reachability into `SignAll.vue` is via self-initiated flows
(`signer/toSignArray` call sites in `ScheduledPayments.vue`, `ScheduledPaymentDetail.vue`,
`Pay.vue`, `Arc200/OptIn.vue`) rather than direct DApp input ‚Äî exploitability depends on whether a
malicious smart contract or a compromised scheduled-payment configuration can cause one of these
flows to construct a rekey transaction inside a batch. **Needs verification** ‚Äî the
transaction-building code for scheduled payments was not traced far enough in this pass to confirm
or rule out attacker reachability with certainty.

**Impact if reachable:** A rekey hidden inside a batch of otherwise-innocuous transactions renders
with zero visual indication in the one screen whose entire purpose is per-transaction review before
bulk-signing.

**Recommendation:** Populate `rekeyTo` on `TransactionTableEntry` the same way
`ConnectRequestsTable.vue` does, and give the column a `#body` template with the same red-flag
treatment `Sign.vue` uses for its single-transaction rekey warning.

### AW-2026-008 ‚Äî WalletConnect v2's built-in DApp-verification signal (`verifyContext`) is received but never surfaced
**Severity:** Medium ¬∑ **Status:** New

**Affected files:** `src/store/wc.ts:204-206, 212-213` (`session_proposal`/`session_request`
handlers commit the whole event object but never read `.verifyContext`), `src/pages/Connect.vue:156-198`
(session-proposal table renders only self-reported `metadata.{name,url,icons,description}`).

**Description:** WalletConnect v2 (via `@reown/walletkit`) attaches a `verifyContext` object to
every session proposal and request, flagging whether the connecting DApp's claimed metadata matches
its actual serving origin or is a known-scam domain (VALID/INVALID/UNKNOWN/SCAM). This wallet
receives that signal on every connection but never reads or displays it ‚Äî a malicious DApp can set
`metadata.name`/`metadata.icons` to impersonate a trusted DApp with no way for the user to detect
it via this app's UI.

**Attacker precondition:** None beyond initiating any WalletConnect pairing.

**Impact:** Weakens phishing resistance for the primary external-input surface; this is the
WalletConnect-recommended anti-phishing feature the app currently omits.

**Recommendation:** Read `verifyContext.verified.validation`/`isScam` off both events, store
alongside the proposal/request, and render a clear trust badge/warning in `Connect.vue`.

### AW-2026-009 ‚Äî Session approval defaults to granting every wallet account across every configured network
**Severity:** Medium ¬∑ **Status:** New

**Affected files:** `src/pages/Connect.vue:417` (`const allAccounts = ref(true);`, default-checked
checkbox), `src/store/wc.ts:351-362` (`approveSession` loops all
`rootState.wallet.privateAccounts` across all networks when true).

**Description:** The default, unmodified "Connect" action authorizes the connecting DApp's session
for every account in the wallet across every configured network, not just the account the user was
viewing. Each transaction still requires an explicit per-transaction Sign click, but this silently
widens the blast radius of any deceptive sign request (e.g. AW-2026-001) from one account to the
entire wallet.

**Recommendation:** Default `allAccounts` to `false`; scope the checkbox with a clearer label
enumerating which accounts will be exposed.

### AW-2026-012 ‚Äî No auto-lock/session timeout; decrypted key material persists indefinitely in memory; dead `auto_lock_off` locale string

### AW-2026-015 ‚Äî No Content-Security-Policy anywhere in the deployment stack
**Severity:** Low‚ÄìMedium ¬∑ **Status:** New

**Affected files:** `index.html` (no CSP meta tag), no `vercel.json` in repo root, K8s ingress
manifests (`k8s/deployment-*.yaml`, no header-injection annotations). GitHub Pages cannot set
custom HTTP headers at all, so that deployment target is structurally unprotected regardless of

uniformly across GH Pages/Vercel/K8s) with at minimum `default-src 'self'; frame-ancestors 'none';
object-src 'none'`, adjusted for the algod/indexer API origins and WalletConnect relay origins
actually used.

### AW-2026-018 ‚Äî Genesis hash/ID shown in WalletConnect detail view but never actively cross-validated against the active network

### AW-2026-019 ‚Äî Unbounded recursion in `getSK` on a multi-hop rekey cycle (DoS)
**Severity:** Low ¬∑ **Status:** New

**Affected file:** `src/store/wallet.ts:730-752` ‚Äî only a direct self-loop
(`rekeyedTo == addr`) is guarded against; a longer cycle (A‚ÜíB‚ÜíA) recorded in local state causes

**Attacker precondition:** Requires an actual on-chain rekey cycle among the user's own imported

party.

**Recommendation:** Track visited addresses and throw on cycle detection instead of recursing
unboundedly.

### AW-2026-021 ‚Äî ARC-14 signed auth token lacks explicit endpoint/audience/timestamp binding
**Severity:** Medium ¬∑ **Status:** New ‚Äî Needs verification (server-side dependent)

**Affected file:** `src/store/arc14.ts:81-141` ‚Äî the signed 0-amount self-payment's only binding to
a "purpose" is a free-text `realm` string in the note field; no explicit destination-URL/audience
claim, no added timestamp/nonce beyond the algod-provided validity-round window (typically several
minutes).

**Description:** Whether this is replayable across services depends entirely on server-side ARC-14
verification logic (outside this repo) actually checking note content against the expected
realm/service and rejecting stale rounds. This codebase adds no additional client-side protection.

**Recommendation:** If not already enforced server-side, bind the note to a concrete audience (full
URL, not an abstract realm name) and use a short first/last-valid window.

### AW-2026-022 ‚Äî Production deploy workflow has no build/test gate of its own
**Severity:** Medium ¬∑ **Status:** New

**Affected file:** `.github/workflows/awallet-main.yml` ‚Äî triggers on every push to `master`; its

deploy step, or confirm/document that `deploy.sh` performs an equivalent gate.

## Positive / no-finding confirmations

Recorded for auditability, per methodology ¬ß2.6 ("do not report a finding you have not personally
traced"), the following were specifically checked and found correct:

- **Signing byte-prefix correctness across every account type** ‚Äî `signBySk`, `signByHd`,
  `signByLedger`, `signByWC1`/`signByWC2`, and all multisig variants in `src/store/signer.ts` were
  each traced to confirm the bytes signed match the bytes the user is shown, including the
  HD-wallet `bytesToSign()` vs. `toByte()` distinction documented in `CLAUDE.md`, and the
  Ledger/WalletConnect paths' correct use of unprefixed bytes (the receiving device/counterparty
  applies the `TX` domain separator itself in both cases, per the standard Ledger-Algorand and
  ARC-0001 integration patterns).
- **HD derivation path** fixed to `m'/44'/283'/account'/0/keyIndex`; `account`/`keyIndex` are always
  sourced from local wallet state or explicit user input at creation time, never from untrusted
  external input.
- **Multisig threshold enforcement** is correctly left to consensus at broadcast time; no
  client-side "partial signature set treated as sufficient" logic was found.
- **Export/backup flows** require password re-entry (`wallet/checkPassword`) before revealing any
  mnemonic/Shamir share/sk-derived data; `backupWallet` exports only the still-AES-encrypted blob.
- **Rekeyed-account resolution** (`syncAccountSigner`, `resolveEnvRekey`) correctly follows on-chain
  `auth-addr` and warns the user when rekey state changes or the resolved signer lacks key material.
- **`v-html` usage** is limited to `Changelog.vue`, whose only input is a compile-time-bundled
  `CHANGELOG.md` file ‚Äî not reachable by runtime/DApp/user input.
- **No open-redirect pattern** in `src/router/index.ts` ‚Äî no route reads a redirect-target query
  param.
- **QR-scanned content** is stripped/validated before use in both `Sign.vue` (non-address
  characters stripped) and `Connect.vue` (used only as a WalletConnect URI input, never rendered as
  HTML/URL).
- **No install-time arbitrary code execution** ‚Äî `pnpm-workspace.yaml`'s `allowBuilds` is narrowly
  scoped to `cypress`/`esbuild`; no `postinstall`/`preinstall` scripts found in the lockfile.
- **Deploy secrets are not exposed to PR-triggered workflows** ‚Äî none of the four GitHub Actions
  workflows use `pull_request_target`; SSH deploy secrets are gated to direct pushes to `master`
  only (branch-protection settings themselves not verifiable from files alone).
- **PWA service-worker config** uses `registerType: "autoUpdate"` with `cleanupOutdatedCaches: true`,
  minimizing the window where a security-fixed build fails to reach a returning user, versus a
  manual-prompt update strategy.
- **Node polyfill/alias config** in `vite.config.ts` uses a narrow, explicit include list, not a
  wildcard; the documented `libsodium-wrappers-sumo` alias points at a fixed local path, not a
  remote source.
