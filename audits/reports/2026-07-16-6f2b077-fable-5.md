This audit covers the 13 commits landed since the prior report (`4483920..6f2b077`). The two
findings the previous audit raised are both confirmed fixed and committed in `57a1002`:
`typescript` was pinned back to `^6.0.3` and the full `pnpm run build` pipeline (vue-tsc

accurate than at the last audit. The bulk of the remaining diff is a visual redesign (auth

Two new findings, neither a fund-loss path. **AW-2026-037 (Medium):** the redesigned FAQ page
renders 50 locale-file answer strings per language through `v-html` — deliberately, because the
answers contain raw `<a>` links — making all 10 `src/locales/*.json` files part of the XSS trust

DApp sessions are no longer received anywhere in the app until the user manually clicks the

behavior, not a vulnerability. No prior open finding regressed.

**Findings this audit:** New: 1 Medium, 1 Low. Fixed since last audit: 2 (AW-2026-035,

## Findings

### AW-2026-037 — FAQ redesign renders locale-file HTML through `v-html`, adding all 10 locale files to the XSS trust boundary

- **Severity:** Medium
- **Status:** New
- **Affected files:** `src/pages/FAQ.vue:65,153-164`, `src/locales/*.json` (all 10,
  `faq.categories.*.a*` keys — 13 embedded `<a href>` tags per file)
- **Description:** The redesigned FAQ page (commit `466ff8f` and successors) renders each of
  its 50 answer strings via `<p class="faq-answer" v-html="renderHtml(item.a)" />`
  (`FAQ.vue:65`). Unlike the `Changelog.vue` case (AW-2026-036, now fixed by escaping), the
  raw-HTML rendering here is *intentional*: answer strings in the locale files deliberately
  embed `<a href="...">` links to Biatec/GitHub/Discord resources, so wholesale escaping cannot
  be applied. The component does escape the one dynamic interpolation (`{name}` →
  `escapeHtml(getWalletBrandName())`, itself a hardcoded constant chosen by hostname), and an
  explanatory comment marks the sink. But the net effect is that every string under
  `faq.categories.*` in all 10 `src/locales/*.json` files is now injected into the DOM as live
  HTML on a public, unauthenticated route. Verified today: all 13 anchors per locale point at
  hardcoded trusted domains (`biatec.io`, `github.com/scholtz/...`, `discord.gg`,
  `github.com/algorandfoundation/...`) and every one carries `target="_blank"
  rel="noopener noreferrer"`; no other HTML tags appear in the answer strings. So there is no
  exploitable payload at this commit.
- **Attacker precondition:** Write/merge access to any of the 10 locale JSON files. This is the
  same precondition class as AW-2026-036, but with a materially wider realistic surface:
  translation updates (e.g. this cycle's Turkish translation commit `466ff8f`) are a common,
  low-scrutiny contribution type, often supplied by community translators and reviewed by
  someone who cannot read the language. A `<img onerror=...>` or a swapped `href` inside a
  Slovak or Turkish answer string would not stand out in a translation diff the way it would in
  a `.vue` file.
- **Impact:** Stored XSS on `/faq` — full page-origin script execution on the same origin that
  holds encrypted wallet blobs in IndexedDB and (if a session is open in another tab) the
  decrypted in-memory signing surface. Also enables phishing-grade content injection (changing
  a trusted support link to an attacker domain survives even an HTML-sanitizing fix if `href`
  values aren't allowlisted). Given the repo-write precondition, Medium rather than High.
- **Recommendation:** Stop treating locale strings as HTML. Preferred: replace the embedded
  anchors with vue-i18n component interpolation (`<i18n-t>` with a `RouterLink`/`<a>` slot per
  link, URLs supplied from a hardcoded constant map in the component, not from the locale
  file), which removes the `v-html` entirely and auto-escapes translated text. If `v-html` must
  stay short-term: run each answer through a strict sanitizer that allowlists only `<a>` with
  `href` matching a fixed set of trusted URL prefixes (and forces
  `rel="noopener noreferrer"`), escaping everything else — and add a CI check (extend
  `scripts/check-locales.ts`) that fails if any locale string contains `<` outside that
  allowlisted pattern, so a malicious translation PR fails the build instead of shipping.

### AW-2026-038 — WalletConnect client no longer initializes in the background; approved-session sign requests silently unreceived until manual init

- **Severity:** Low
- **Status:** New
- **Affected files:** `src/App.vue` (commit `9471074`, removal), `src/pages/Connect.vue:136-140,521-523`
- **Description:** Commit `9471074` removed `App.vue`'s `initWc()` bootstrap — the
  mounted-hook + `walletIsOpen` watcher that dispatched `wc/init` as soon as the wallet was
  opened. The removed code carried a comment explicitly documenting its purpose: to receive
  incoming signature requests from already-approved DApp sessions (and update the navbar badge)
  regardless of which page the user is on. At this commit, the only remaining `wc/init`
  dispatch site in the app is `Connect.vue:522`, reachable only by navigating to
  `/account/connect` and clicking the "Init WC" button (`Connect.vue:137` — the button is only
  rendered while `wc.web3wallet` is null, i.e. init never happens automatically even on that
  page). Consequence: after a page reload or fresh login, a DApp with an approved session that
  sends a `session_request` gets no response and the user sees nothing, anywhere, until they
  manually re-initialize — the wallet appears paired (the session persists in WalletConnect
  storage) but is functionally deaf.
- **Attacker precondition:** None — this is a self-inflicted functional regression, not an
  attacker-triggered condition. It marginally *reduces* remote attack surface (the WalletKit
  client and its relay socket aren't running unless explicitly started).
- **Impact:** No direct fund/key exposure. Two second-order concerns: (1) silent failure of a
  security-relevant flow — a user who believes their wallet is connected may retry pairing
  repeatedly or approve a fresh session under time pressure (approval fatigue is how
  wrong-session/wrong-DApp mistakes happen); (2) the regression contradicts the codebase's own
  documented intent, suggesting the removal may have been accidental (the commit message says
  "refactor", and no replacement init path was added elsewhere).
- **Recommendation:** If the always-on listener is still desired behavior, restore the
  bootstrap (in `App.vue` or `layouts/Main.vue`'s wallet-open branch) — dispatching `wc/init`
  when `wallet.isOpen` becomes true, guarded by `if (state.wc.web3wallet) return` as before. If
  the removal was deliberate (e.g. to avoid opening a relay socket without user intent, which
  is a defensible privacy/surface choice), make the tradeoff visible: auto-init on
  `/account/connect` page mount instead of behind a button, and show a clear "not listening for
  DApp requests" indicator wherever active sessions are listed, so the deaf-but-paired state is
  never silent. Either way, record the decision in CLAUDE.md, since the deleted comment was the
  only documentation of the previous behavior.

  (vue-tsc `--noEmit` + `vite build`) at `6f2b077`: it completes cleanly, producing `dist/` and
  the PWA service worker (69 precache entries). `docker/Dockerfile`'s `RUN pnpm run build` step
  is therefore unblocked.
- **AW-2026-036** — unescaped markdown/interpolation into `v-html` in `Changelog.vue` and
  `Footer.vue`. Fixed in commit `57a1002`, re-verified in source at `6f2b077`:
  `Changelog.vue:47-51` now runs `escapeHtml` (escaping `&`, `<`, `>`) on every line *before*
  any tag is constructed in `renderInline`; `Footer.vue:76-87` escapes the `name`, `envStatus`,
  and link-label values interpolated into `brandLineHtml`, with a comment referencing the audit
  finding so the constraint survives future edits. Note: the same *pattern* was independently
  reintroduced in a third location this cycle by the FAQ redesign — tracked as its own finding
  (AW-2026-037) rather than a regression of this one, because the FAQ sink renders
  intentionally-HTML locale content and needs a different fix than escaping.
