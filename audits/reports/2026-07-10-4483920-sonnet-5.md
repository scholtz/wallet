## Findings

### AW-2026-035 â€” `pnpm run build` fails at HEAD: `typescript@7.0.2` bump breaks `vue-tsc` and `tsc`

- **Severity:** High
- **Status:** New
- **Affected files:** `package.json:127` (`typescript: ^7.0.2`), `tsconfig.json:9,17,20,22`,
  `docker/Dockerfile` (`RUN pnpm run build`)
- **Description:** Commit `4483920` bumped the `typescript` devDependency from `^6.0.3` to
  `^7.0.2` (alongside an unrelated alphabetical reorder of other deps). Running the documented
  build command fails in two independent ways:
  1. `pnpm run build` (`vue-tsc --noEmit && vite build`) throws
     `Error [ERR_PACKAGE_PATH_NOT_EXPORTED]: Package subpath './lib/tsc' is not defined by "exports"
     in .../typescript/package.json` â€” `vue-tsc@3.3.7` still does `require.resolve("typescript/lib/tsc")`,
     a subpath TypeScript 7's `package.json` `exports` map no longer publishes.
  2. Independently, `pnpm run check-typescript-errors-tsc` (bare `npx tsc --noEmit`) fails with:
     `tsconfig.json(9,25): error TS5108: Option 'moduleResolution=node10' has been removed`,
     `tsconfig.json(17,5): error TS5102: Option 'baseUrl' has been removed`, plus two
     `TS5090: Non-relative paths are not allowed` errors on the `paths` entries `baseUrl` used to
     anchor. TypeScript 7 dropped both compiler options this repo's `tsconfig.json` was built
     around (including the `moduleResolution: "node"`/classic-resolution workaround CLAUDE.md
     documents as load-bearing for the `@primeuix/themes` deep-import `paths` alias).
  A bare `npx vite build` (skipping the `vue-tsc` gate) still succeeds and produces a working
  `dist/`, confirming this is a type-checking-toolchain break, not a runtime/bundling break â€” but
  it is not what `docker/Dockerfile`'s `RUN pnpm run build` step runs, so a Docker image build from
  this commit onward would fail closed at that line.
- **Attacker precondition:** None â€” this is a build-tooling regression, not an attacker-triggered
  condition.
- **Impact:** Not itself a path to fund/key loss. Impact is availability of the fix pipeline: no
  Docker image (and therefore no K8s/GitHub Pages/Vercel deploy that goes through `pnpm run build`)
  can be produced from this commit or later until either `typescript` is pinned back to a `vue-tsc`-
  and-`tsconfig`-compatible major version, or `vue-tsc`/`tsconfig.json` are updated for TS7's new
  `exports` map and removed options. Because this blocks *all* deploys, not just this feature, it
  would equally block landing a fix for a genuinely critical finding until resolved â€” the
  "fail-closed" direction is safe, but the delay itself is a real availability risk for a wallet
  that ships fixes to a static SPA with no other patch mechanism.
- **Recommendation:** Either pin `typescript` to the last `6.x` released version that `vue-tsc@3.3.7`
  and the current `tsconfig.json` support, or do the paired upgrade properly: bump `vue-tsc` to a
  version compatible with TS7's `exports` map, remove/replace `moduleResolution: "node10"` and
  `baseUrl` in `tsconfig.json` with their TS7-supported equivalents, and re-verify the `@primeuix/themes`
  deep-import `paths` workaround (CLAUDE.md's existing note about not switching to `"bundler"`
  resolution) still holds under whatever resolution mode replaces it. Add a CI check that actually
  runs `pnpm run build` (see AW-2026-022, still open) so a break like this is caught before merge
  rather than by the next audit.

### AW-2026-036 â€” New `/changelog` page renders unescaped markdown via `v-html`; related `v-html` sink in the footer

- **Severity:** Medium
- **Status:** New
- **Affected files:** `src/pages/Changelog.vue:12,41-78`, `src/components/Footer.vue:7,71-82`
- **Description:** `Changelog.vue` (new this cycle) imports `CHANGELOG.md` as a raw string and
  converts it to HTML with a hand-written `toHtml`/`renderInline` function before rendering via
  `v-html="html"`. The converter interpolates each Markdown line directly into a template literal
  (e.g. `` `<h2>${renderInline(line.slice(3))}</h2>` ``, `` `<li>${renderInline(line.slice(2))}</li>` ``)
  with no HTML-entity escaping of `<`, `>`, or `&` in the source text â€” only backtick-delimited
  inline code gets special handling, and that handling (wrapping in `<code>`) doesn't escape its
  contents either. `CHANGELOG.md` currently contains no angle brackets (verified via
  `grep -n '[<>&]' CHANGELOG.md`), so there is no exploitable payload today. But the file is
  ordinary Markdown documentation maintained by repo contributors, and PR review of a changelog
  bullet point does not typically get the same scrutiny as a `.vue`/`.ts` diff â€” a bullet like
  `- Fixed the <Password> feedback flag` (a completely natural thing to write, referencing the
  actual `<Password :feedback>` component name used elsewhere in this codebase) or a pasted
  `<img onerror=...>` would render as live HTML/JS on the public `/changelog` route for every
  visitor, no login required. Separately, `Footer.vue`'s new `brandLineHtml` computed property
  builds `<a href="...">` strings via a local `link()` helper and feeds the assembled string through
  `this.$t(...)` interpolation into `v-html`. This one is currently safe by construction â€” every
  interpolated value (`getWalletBrandName()`, the three hardcoded URLs, `envStatus`, which is only
  ever set to one of five fixed literal strings from a `switch` over `config.env`) is fully
  app-controlled, not user- or DApp-supplied â€” but it establishes the same unescaped-string-into-`v-html`
  pattern as a second sink, and a future change that threads any less-trusted value (e.g. a
  configurable brand name, or a custom RPC/indexer label) through the same helper would inherit the
  vulnerability silently.
- **Attacker precondition:** For `Changelog.vue`: write/merge access to `CHANGELOG.md` (i.e., a
  malicious or compromised contributor/maintainer), or a future feature that populates any part of
  the rendered content from external input. For `Footer.vue`: none currently exploitable; flagged as
  a latent pattern.
- **Impact:** Stored XSS on `/changelog`, a public, unauthenticated, always-reachable route
  (`layouts/Public.vue`) â€” full page-origin script execution, which for this app means an attacker
  script could reach the very same origin that holds encrypted wallet blobs in IndexedDB and, if a
  wallet session happens to be open in another tab, the decrypted signing/state surface. Given the
  precondition (repo write access) this is not a remote-anonymous-attacker path, hence Medium rather
  than Critical/High.
- **Recommendation:** Escape `<`, `>`, and `&` in `renderInline`/`toHtml` before wrapping in tags
  (or switch to a minimal, well-tested Markdown-to-HTML library that escapes by default), or avoid
  `v-html` entirely by parsing `CHANGELOG.md` into a small structured AST and rendering it with real
  Vue template bindings (which auto-escape). For `Footer.vue`, either drop `v-html` in favor of an
  i18n message with `<i18n-t>`-style slot interpolation (PrimeVue/vue-i18n both support rendering
  translated text with embedded components/links without raw HTML strings), or keep `v-html` but add
  a code comment / lightweight escaping helper so the next person who threads a new value through
  `brandLineHtml` doesn't unknowingly reintroduce the `Changelog.vue` issue.

## Re-verified / corrected from prior audit

- **AW-2026-012** ("No auto-lock/session timeout; keys persist indefinitely in memory") â€” **status
  corrected, remains Open but the description was inaccurate.** `src/components/Footer.vue`
  (unchanged logic this cycle, only its template markup was refactored) does implement a real
  session-expiry mechanism: `state.wallet.time` is set on login, a 1-second interval in `Footer.vue`
  computes elapsed time against a 5-minute (`SESSION_TIMEOUT_MS`) budget, shows a warning dialog at
  30 seconds remaining, and calls `wallet/logout` â€” which clears `state.privateAccounts` and the
  derived-key cache (AW-2026-027 fix) â€” the instant the budget is exceeded. So keys do **not**
  persist "indefinitely." The residual gap: (1) the "Continue Session" button (`continueSession` â†’
  `wallet/prolong` â†’ `state.time = new Date()`) requires **no password re-entry**, so anyone with
  momentary access to an already-open, still-unlocked tab during the 30-second warning window can
  extend the session indefinitely without ever learning the password; (2) `prolong` is only
  triggered by that explicit button (and, incidentally, by `addArc200Asset`) â€” it is a flat 5-minute
  budget from last-prolong, not a real mouse/keyboard idle detector, so a genuinely active user who
  never touches those two triggers is force-logged-out every 5 minutes regardless of activity (a
  UX rough edge, not a security gap). Net effect: this is a working timeout with a weak
  re-authentication step at renewal, not an absent timeout. Recommend requiring the password (or at
  minimum a shorter, harder-to-catch renewal window) on "Continue Session," rather than treating this
  as the same finding as "no timeout at all." Severity unchanged (Medium â†’ arguably now Low given a
  real control exists); left as Medium pending the fix to close the re-auth gap.
